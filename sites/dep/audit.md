## 2. Анализ логики работы

### 2.1 Аутентификация и авторизация

**Механизм:**
- Пользователи хранятся в `.env` в формате `USERS=login:password,login2:password2`
- Пароли хранятся в **открытом виде** (plain text)
- Сессионная модель PHP (`$_SESSION['user']`)
- Проверка прав на уровне проекта (`ALLOWED_USERS`)
- Отдельная проверка прав на PROD-деплой (`PROD_ALLOWED_USERS`)

**Логика доступа:**
```
Global USERS → Project ALLOWED_USERS → PROD_ALLOWED_USERS (если target=prod)
```

### 2.2 Получение списка веток

**Текущая реализация:**
- AJAX-запрос к `ajax.php?action=get_branches`
- Прямой запрос к GitHub API v3 (REST)
- Для каждой ветки — отдельный запрос за деталями коммита
- Сортировка по дате коммита (от новых к старым)

**Проблема производительности:**
```php
// N веток = N+1 HTTP-запросов к GitHub API
foreach ($branchesList as $branchData) {
    $commitResponse = githubApiRequest($commitUrl, $githubToken);
    // ...
}
```

При 50 ветках — 51 запрос, что может занимать 5-10 секунд.

### 2.3 Процесс деплоя

**Последовательность операций в `deploy_remote.sh`:**

```bash
1. Проверка ALLOWED_USERS (доступ к проекту)
2. Проверка PROD_ALLOWED_USERS (если IS_PROD=1)
3. cd $FOLDER
4. git fetch origin
5. git reset --hard      # ⚠️ Удаляет все локальные изменения
6. git checkout $BRANCH
7. git pull origin $BRANCH
```

**Канал передачи логов:**
- Логи пишутся в `lastlog.log` на стороне Windows
- Frontend опрашивает лог каждые 1200 мс (polling)
- **Проблема:** нет разделения логов между одновременными деплоями

## 3. Аудит безопасности

### 3.1 КРИТИЧЕСКИЕ уязвимости (Critical)

| # | Уязвимость | Риск | Файл | Описание |
|---|------------|------|------|----------|
| 1 | **Пароли в plain text** | 🔴 Critical | `.env`, `index.php` | `USERS=admin:12345` — пароли хранятся открыто |
| 2 | **SSH-пароль в конфиге** | 🔴 Critical | `projects/*.env` | `SSH_PASS=ca9isterint3` — пароль сервера в открытом виде |
| 3 | **GitHub PAT в конфиге** | 🔴 Critical | `projects/*.env` | Токен `github_pat_...` скомпрометирован при доступе к файлу |
| 4 | **Передача пароля SSH в командной строке** | 🔴 Critical | `ajax.php` | `-pw "$sshPass"` — видно в `ps aux` других пользователей |
| 5 | **Отсутствие rate limiting** | 🟠 High | `ajax.php` | Брутфорс логина без ограничений |
| 6 | **session fixation** | 🟠 High | `index.php` | Нет `session_regenerate_id()` после логина |

### 3.2 Уязвимости высокого риска (High)

| # | Уязвимость | Файл | Описание |
|---|------------|------|----------|
| 7 | **SQL Injection (потенциальная)** | `ajax.php` | Хотя SQL нет, но парсинг `.env` через `explode` может сломаться на спецсимволах |
| 8 | **XSS в логах** | `index.php`, `app.js` | `htmlspecialchars()` применяется, но лог может содержать JS-инъекции из Git |
| 9 | **Path Traversal** | `ajax.php` | `$project` из POST не валидируется на `/` и `..` |
| 10 | **Command Injection** | `ajax.php` | `escapeshellarg()` используется, но `$remoteCommand` формируется конкатенацией |
| 11 | **Отсутствие HTTPS проверки** | `ajax.php` | `CURLOPT_SSL_VERIFYPEER = false` — MITM-атака на GitHub API |
| 12 | **Hardcoded credentials в коде** | `deploy_remote.sh` | Передача `$SSH_PASS` через параметры скрипта |

### 3.3 Уязвимости среднего риска (Medium)

| # | Уязвимость | Файл | Описание |
|---|------------|------|----------|
| 13 | **Отсутствие CSRF-защиты** | `ajax.php` | Нет токенов в POST-запросах |
| 14 | **Слабая изоляция сессий** | `index.php` | Нет `session.cookie_httponly`, `session.cookie_secure` |
| 15 | **Логирование чувствительных данных** | `lastlog.log` | В лог попадают имена пользователей и пути |
| 16 | **Небезопасное удаление файлов** | — | `git reset --hard` может удалить важные данные |
| 17 | **Отсутствие аудита действий** | — | Нет журнала кто/когда/что задеплоил |

### 4.2 Преимущества текущей системы

1. **Простота развёртывания** — не требует Composer, Ruby, Node.js
2. **Веб-интерфейс** — доступно любому пользователю через браузер
3. **Минимальные требования** — работает на shared-хостинге с PHP
4. **Прозрачность** — весь код виден, легко модифицировать

### 4.3 Недостатки

1. **Нет zero-downtime деплоя** — простой сайта во время деплоя
2. **Нет отката** — нельзя быстро вернуться к предыдущей версии
3. **Нет хуков** — нельзя запустить тесты, миграции, clear cache
4. **Один файл логов на все деплои** — конфликты при параллельных запусках
5. **Синхронный деплой** — блокировка интерфейса на время выполнения

### 5.2 Конкретные рефакторинги

#### 5.2.1 Безопасная работа с SSH

**Было:**
```php
$cmd = "\"$plink\" -ssh -batch -pw \"$sshPass\" $sshUser@$sshHost $remoteCommand";
```

**Стало:**
```php
// Использовать SSH-ключи с passphrase
$cmd = "\"$plink\" -ssh -i \"{$sshKeyPath}\" -batch $sshUser@$sshHost $remoteCommand";

// Или использовать SSH-агент (Pageant)
```

#### 5.2.2 Улучшение работы с GitHub API

**Было:**
```php
foreach ($branchesList as $branchData) {
    $commitResponse = githubApiRequest($commitUrl, $githubToken);
}
```

**Стало (GraphQL один запрос):**
```graphql
query {
  repository(owner: "sergey0002", name: "noff-site") {
    refs(refPrefix: "refs/heads/", first: 50) {
      nodes {
        name
        target {
          ... on Commit {
            oid
            committedDate
            message
            author { name }
          }
        }
      }
    }
  }
}
```

#### 5.2.3 Zero-downtime деплой

**Схема:**
```bash
# Вместо прямого деплоя в www
/releases/
  ├── 20260311_120000/     # Новая версия
  ├── 20260310_150000/     # Предыдущая версия
  └── current -> /releases/20260310_150000  # symlink

# Алгоритм:
# 1. git clone в /releases/TIMESTAMP/
# 2. git checkout $BRANCH
# 3. Запуск тестов/миграций
# 4. mv current -> /releases/TIMESTAMP/ (атомарно)
# 5. Хранить последние N версий для отката
```

#### 5.2.4 Асинхронный деплой

**Было:**
```php
shell_exec($cmd); // Блокирует на 30-60 секунд
echo json_encode(['status' => 'ok']);
```

**Стало (очередь задач):**
```php
// 1. Сохранить задачу в БД/файл
$taskId = uniqid('deploy_');
file_put_contents("storage/queue/{$taskId}.json", json_encode([
    'branch' => $branch,
    'target' => $targetName,
    'user' => $user,
    'status' => 'pending'
]));

// 2. Запустить фоновый процесс
shell_exec("php worker.php {$taskId} > storage/logs/{$taskId}.log 2>&1 &");

// 3. Вернуть ID задачи клиенту
echo json_encode(['status' => 'ok', 'task_id' => $taskId]);

// 4. Frontend polling по task_id
```
