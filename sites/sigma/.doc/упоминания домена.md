# Упоминания домена `xdemo` в `sites/hotel`

Собрано поиском по проекту `sites/hotel`.

Команда:

```powershell
rg -n "xdemo|xdemo\.m2profi\.pro|sites/xdemo" sites/hotel -g '!sites/hotel/sahmatka/php-error.log' -g '!*.csv' -g '!sites/hotel/sahmatka/actions/todo.txt'
```

Примечание:
- В список попали и рабочие URL, и комментарии/подписи файлов, и настройки БД `m2profi_xdemo`.
- `php-error.log`, `*.csv` и `actions/todo.txt` исключены как шумные файлы.

## Цель выноса домена в конфиг

Сейчас домен размазан по PHP, шаблонам, JS, ajax-ссылкам, iframe-ссылкам, картинкам, фидам и внешним виджетам. Из-за этого перенос `hotel` на другой домен требует ручной правки десятков файлов и легко приводит к смешиванию окружений, когда страница `hotel` продолжает ходить данными в `xdemo`.

Нужная конечная схема:

- Домен и базовые URL задаются в одном месте в `config.php`.
- PHP-код строит ссылки только через helper-функции или значения из конфига.
- Внутренний JS не содержит домен строкой, а получает готовые URL из PHP.
- Внешние виджеты не содержат домен внутри собственного `.js`, а получают его при подключении.
- `rg -n "xdemo|xdemo\\.m2profi\\.pro"` после миграции должен находить только исторические комментарии или архивные файлы.

## Что именно хранить в конфиге

Лучше хранить не один `domain`, а набор нормализованных базовых адресов, потому что в проекте используются разные типы ссылок.

Минимальный набор:

```php
$GLOBALS['config']['site'] = [
    'scheme' => 'https',
    'host' => 'hotel.m2profi.pro',
    'base_url' => 'https://hotel.m2profi.pro',
    'sahmatka_url' => 'https://hotel.m2profi.pro/sahmatka',
    'ajax_router' => 'https://hotel.m2profi.pro/sahmatka/ajax_router.php',
    'iframe_router' => 'https://hotel.m2profi.pro/sahmatka/iframe_router.php',
    'ajax_actions' => 'https://hotel.m2profi.pro/sahmatka/ajax_actions.php',
    'assets_url' => 'https://hotel.m2profi.pro',
];
```

Если домен иногда должен определяться автоматически, допустим fallback:

```php
$host = $_SERVER['HTTP_HOST'] ?? 'hotel.m2profi.pro';
$scheme = (!empty($_SERVER['HTTPS']) && $_SERVER['HTTPS'] !== 'off') ? 'https' : 'http';
```

Но для прода лучше оставить возможность жестко переопределить host в конфиге, чтобы CLI, cron и прокси не ломали генерацию ссылок.

## Какие helper-функции нужны в PHP

Вместо прямой склейки строк по всему проекту лучше сделать 3-5 простых функций и потом переводить код на них.

Пример набора:

```php
function site_base_url() {
    return $GLOBALS['config']['site']['base_url'];
}

function site_sahmatka_url($path = '') {
    return rtrim($GLOBALS['config']['site']['sahmatka_url'], '/') . '/' . ltrim($path, '/');
}

function site_ajax_router($query = '') {
    $url = $GLOBALS['config']['site']['ajax_router'];
    return $query ? $url . (strpos($url, '?') === false ? '?' : '&') . ltrim($query, '?&') : $url;
}

function site_iframe_router($query = '') {
    $url = $GLOBALS['config']['site']['iframe_router'];
    return $query ? $url . (strpos($url, '?') === false ? '?' : '&') . ltrim($query, '?&') : $url;
}

function site_asset_url($path = '') {
    return rtrim($GLOBALS['config']['site']['assets_url'], '/') . '/' . ltrim($path, '/');
}
```

Этих функций достаточно, чтобы убрать почти все вхождения домена из:

- контроллеров `fw/controllers`
- шаблонов `fw/templates`
- `user.php`
- `ajax_actions.php`
- `actions/*.php`

## Как передавать домен в обычный JS

Для внутреннего JS домен не должен быть зашит в `.js` файле. Правильнее всего отдавать в шаблоне один глобальный объект конфигурации.

Точка входа:

- `sites/hotel/sahmatka/template/default/in_head.php`

Туда можно вывести:

```php
<script>
window.M2PROFI_CONFIG = {
    baseUrl: <?= json_encode(site_base_url()) ?>,
    sahmatkaUrl: <?= json_encode(rtrim($GLOBALS['config']['site']['sahmatka_url'], '/')) ?>,
    ajaxRouter: <?= json_encode($GLOBALS['config']['site']['ajax_router']) ?>,
    iframeRouter: <?= json_encode($GLOBALS['config']['site']['iframe_router']) ?>,
    ajaxActions: <?= json_encode($GLOBALS['config']['site']['ajax_actions']) ?>,
    assetsUrl: <?= json_encode(rtrim($GLOBALS['config']['site']['assets_url'], '/')) ?>
};
</script>
```

После этого любой JS должен собирать URL только так:

```js
const cfg = window.M2PROFI_CONFIG;
const url = cfg.ajaxRouter + '?ctr=rentobjects&act=sel_street';
```

Или чуть чище:

```js
function m2Url(path) {
    return window.M2PROFI_CONFIG.sahmatkaUrl + '/' + String(path).replace(/^\/+/, '');
}
```

Тогда в коде:

```js
sendAjaxForm('street', 'rentsearch', m2Url('ajax_router.php?ctr=rentobjects&act=sel_street'), 0);
```

## Как передавать домен в виджеты

С виджетами отдельная история: они могут жить вне `sahmatka` и подключаться на стороннем сайте, поэтому `window.M2PROFI_CONFIG` там может не существовать.

Надёжный алгоритм для виджетов:

1. Убрать домен из `wiget_*.js`.
2. Передавать `baseUrl` через inline-конфиг перед подключением виджета или через `data-*` у тега `<script>`.
3. Внутри виджета читать настройки из `document.currentScript` или из заранее созданного `window.M2WidgetConfig`.

Предпочтительный формат подключения:

```html
<script>
window.M2WidgetConfig = {
    baseUrl: 'https://hotel.m2profi.pro',
    sahmatkaUrl: 'https://hotel.m2profi.pro/sahmatka'
};
</script>
<script src="https://hotel.m2profi.pro/wiget_rent.js"></script>
```

Либо через `data-*`:

```html
<script
    src="https://hotel.m2profi.pro/wiget_rent.js"
    data-base-url="https://hotel.m2profi.pro"
    data-sahmatka-url="https://hotel.m2profi.pro/sahmatka"></script>
```

Внутри самого виджета:

```js
var current = document.currentScript;
var cfg = window.M2WidgetConfig || {};
var baseUrl = cfg.baseUrl || (current && current.dataset.baseUrl) || '';
var sahmatkaUrl = cfg.sahmatkaUrl || (current && current.dataset.sahmatkaUrl) || (baseUrl ? baseUrl + '/sahmatka' : '');
```

После этого все URL внутри виджета строятся от `baseUrl` или `sahmatkaUrl`, а не от `https://xdemo.m2profi.pro/...`.

## Как мигрировать существующие JS-файлы

Лучше идти по слоям, а не массовым search/replace по всему проекту.

Порядок:

1. Ввести конфиг и helper-функции в PHP.
2. Вынести `window.M2PROFI_CONFIG` в общий шаблон.
3. Перевести внутренние JS-файлы, которые работают только внутри `sahmatka`.
4. Отдельно перевести публичные виджеты `wiget_*.js`.
5. Потом перевести шаблоны и контроллеры, где ссылки печатаются прямо в HTML.
6. В конце перевести фиды, cron и служебные скрипты.

Приоритет по JS:

- `sites/hotel/sahmatka/fw/controllers/ctr__rentobjects.php`
- `sites/hotel/sahmatka/template/default/js/myfw_iframe.js`
- `sites/hotel/wiget_rent.js`
- `sites/hotel/wiget_rent_v2.js`
- `sites/hotel/wiget_catalog.js`
- `sites/hotel/wiget_home.js`
- `sites/hotel/wiget_home2.js`

## Как избежать поломки ajax и iframe

Нельзя смешивать абсолютные и относительные пути бессистемно. Надо заранее разделить сценарии.

Правило:

- Для внутренней админки и страниц на том же домене лучше использовать либо относительные пути, либо URL из `window.M2PROFI_CONFIG`.
- Для внешних виджетов и iframe лучше использовать абсолютные URL из конфига.
- Для картинок, CSS и `script.src` в виджетах тоже нужен единый `assetsUrl`.

Практический шаблон:

- внутренний код: `ajax_router.php?...` или `cfg.ajaxRouter + '?...'`
- внешний код: `cfg.sahmatkaUrl + '/ajax_router.php?...'`

## Что делать с CSS, картинками и фидами

Отдельно проверить и перевести:

- `background-image: url(...)`
- `img src="https://xdemo..."`
- `@import "https://xdemo..."`
- `script.src` и `style.href`
- XML/CSV/Avito/Yandex/Domclick feed URLs
- редиректы и письма, где домен показывается пользователю текстом

Это важно, потому что часть проекта использует `base_url`, а часть не только ходит в API, но и печатает публичные ссылки для клиентов.

## Рекомендуемая стратегия замены

Самый безопасный путь такой:

1. Сначала добавить конфиг и helper-функции без замены старого кода.
2. Перевести 1 модуль целиком, например `rentobjects`.
3. Проверить, что page, ajax, iframe и widget этого модуля работают уже без `xdemo`.
4. Затем перевести `parking`.
5. Затем `apartments`, `zapis`, `homeseditor`, фиды.
6. После каждой итерации прогонять поиск `rg -n "xdemo|xdemo\\.m2profi\\.pro" sites/hotel`.

## На что обратить особое внимание

- `config.php` сейчас содержит не только домен, но и БД `m2profi_xdemo`. Это отдельный вопрос, его не стоит смешивать с URL-миграцией.
- `cron`, `feed` и `import` файлы могут работать без `HTTP_HOST`, поэтому для них обязателен явный домен из конфига.
- Если виджет встраивается на сторонний сайт, относительные пути там не сработают, только абсолютные.
- Если один и тот же JS-файл используется и внутри сайта, и как внешний виджет, лучше разделить эти сценарии, а не пытаться держать одну хрупкую универсальную строку.
- Комментарии с `sites/xdemo2/...` можно не трогать на первом этапе, если цель именно функциональный перенос домена.
