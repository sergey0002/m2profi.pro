# План рефакторинга проекта

## 1. Общие цели рефакторинга

*   Повышение читаемости и поддерживаемости кода.
*   Улучшение производительности и отзывчивости интерфейса.
*   Повышение безопасности приложения.
*   Обеспечение отказоустойчивости критически важных функций, таких как добавление комментариев.
*   Упрощение тестирования и расширения функционала.
*   Разделение ответственности между клиентской и серверной частями.

## 2. Структурирование JavaScript и вынесение статического JS

### 2.1. Разделение кода и модульность

**Текущее состояние:** Большая часть JavaScript-логики находится непосредственно в `dashboard.php` и в `temp_function.js`. Код не структурирован по модулям, что затрудняет его поддержку и расширение.

**Предлагаемые изменения:**

*   **Консолидация:** Объединить существующий JavaScript в меньшее количество файлов для упрощения. Например:
    *   `js/main.js`: Основной файл, содержащий всю логику взаимодействия с DOM, обработчики событий, AJAX-запросы, логику управления комментариями и код для мобильного меню из `access/template/header.php`.
    *   `js/utils.js`: Для общих вспомогательных функций и констант, которые могут быть переиспользованы.
*   **Без ES Modules/Сборщиков:** В текущем проекте нет необходимости в сложных модульных системах или сборщиках. Использовать стандартные скрипты.

### 2.2. Вынесение статического JS

**Текущее состояние:** Статический JavaScript смешан с динамически генерируемым PHP-кодом в `dashboard.php`.

**Предлагаемые изменения:**

*   **Отдельные файлы:** Переместить весь статический JavaScript в отдельные файлы в директории `js/`.
    *   `js/constants.js`: Для таких констант, как `rowcolors` (сейчас генерируется PHP в `dashboard.php`), `event_type_ids` (если нужны на клиенте).
    *   `js/utils.js`: Для общих вспомогательных функций, таких как `renderEvents` (сейчас в `dashboard.php` и `temp_function.js`).
*   **Подключение:** Подключать эти файлы в `dashboard.php` через теги `<script src="path/to/file.js" defer></script>` для асинхронной загрузки и выполнения после парсинга HTML.

### 2.3. Организация файлов

*   Создать новую директорию `js/` в корне проекта для всех JavaScript-файлов.

**Пример структуры файлов:**
```
.
├── config.php
├── dashboard.php
├── js/
│   ├── main.js
│   └── utils.js
├── temp_function.js (будет рефакторизован и его код распределен по новым файлам)
└── ...
```

### 2.4. Управление AJAX-запросами и обратная связь с пользователем

**Цель:** Обеспечить единый механизм обработки AJAX-запросов, визуальной индикации их выполнения и уведомления пользователя о результате, избегая блокирующих `alert()`.

**Предлагаемые изменения:**

*   **Очередь AJAX-запросов (в `js/main.js`):**
    *   Реализовать механизм очереди для AJAX-запросов, чтобы предотвратить одновременную отправку множества запросов и управлять их порядком.
    *   Использовать промисы для обработки ответов.
    *   Функция `enqueueRequest(requestFunction)`: добавляет запрос в очередь и выполняет его, когда предыдущие запросы завершены.

*   **Единый индикатор прогресса (в `js/main.js`):**
    *   Создать элемент UI (например, тонкая полоса в верхней части экрана или спиннер), который будет активироваться при начале AJAX-запроса и деактивироваться по его завершении.
    *   `showProgressBar()` и `hideProgressBar()`: управляют видимостью индикатора.
    *   Связать с очередью запросов: индикатор показывается, когда есть активные запросы в очереди, и скрывается, когда очередь пуста.

*   **Система уведомлений (в `js/main.js`):**
    *   Создать неблокирующую систему уведомлений (например, всплывающие сообщения "тосты" или баннеры) для отображения успеха или ошибки.
    *   `showNotification(message, type)`: принимает сообщение и тип (`success`, `error`, `info`).
    *   Уведомления должны автоматически исчезать через несколько секунд.
    *   **Строгое правило:** Уведомление и обновление фронтенда происходит только после получения успешного ответа от сервера о сохранении данных.

## 3. Обработка ошибок отправки комментариев (без использования LocalStorage)

### 3.1. Механизм обработки ошибок и повторной отправки

**Цель:** Информировать пользователя о проблемах с отправкой комментария и предоставить возможность повторной попытки, не используя `localStorage` для постоянного хранения.

**Реализация (в `js/main.js`):**

*   **Перехват ошибок AJAX-запроса:**
    *   Когда AJAX-запрос на отправку комментария завершается ошибкой (например, сетевая ошибка, таймаут, ошибка сервера), функция `sendComment()` должна возвращать ошибку или специальный статус.
*   **Индикация ошибки и запрос повтора:**
    *   При получении ошибки от `sendComment()`:
        *   Показать пользователю неблокирующее сообщение (например, через `showNotification('Проверьте соединение с интернетом.', 'error')`).
        *   Предложить стандартный диалог подтверждения (`confirm`) с вопросом "Попробовать повторно?".
        *   Если пользователь подтверждает:
            *   Повторить вызов `sendComment()` с теми же данными.
            *   Возможно, обновить UI, чтобы показать, что комментарий снова "в ожидании".
        *   Если пользователь отказывается:
            *   Очистить поле ввода комментария.
            *   Возможно, показать сообщение об отмене отправки.

### 3.2. Визуальная индикация для пользователя

**Реализация (в `js/main.js`):**

*   **Сообщения об ошибке:** Использовать систему уведомлений (уже описанную в разделе 2.4) для отображения сообщения "Проверьте соединение с интернетом.".
*   **Диалог подтверждения:** Использовать стандартный `confirm()` или кастомный модальный диалог для запроса повторной отправки.
*   **Состояние ввода:** После неудачной отправки поле ввода комментария должно оставаться заполненным, чтобы пользователь мог легко повторить попытку или скопировать текст, если он решит не отправлять его снова.

## 4. OOP-рефакторинг (предыдущий план)

**Цель:** Переход к объектно-ориентированной парадигме для улучшения структуры и модульности кода PHP.

**Шаги:**

*   **Классы для сущностей:**
    *   `DatabaseManager` (в `php/DatabaseManager.php`): Для управления подключением к БД и выполнения запросов. Будет инкапсулировать `PDO` объект. Методы: `query`, `prepare`, `execute`.
    *   `CommentService` (в `php/CommentService.php`): Для бизнес-логики, связанной с комментариями (добавление, получение). Методы: `addComment(postId, userId, commentText, color)`, `getCommentsByPostId(postId)`.
    *   `CallerService` (в `php/CallerService.php`): Для логики, связанной с записями `caller` (обновление цвета, получение телефонов). Методы: `updateCallerColor(callerId, color, userId)`, `getCallerPhones(callerId)`, `logPhoneView(callerId, userId)`.
    *   `AuthService` (в `php/AuthService.php`): Для управления сессиями и аутентификацией. Методы: `isLoggedIn()`, `getCurrentUserId()`.
    *   `EventLogger` (в `php/EventLogger.php`): Для логирования событий. Методы: `logEvent(datetime, callerId, eventType, comment, userId)`.
    *   `Utils` (в `php/Utils.php`): Для общих вспомогательных функций, таких как `formatPhoneNumber`.
*   **Рефакторинг существующих PHP-файлов:**
    *   **Единая точка входа:** Все AJAX-обработчики (`add_comment.php`, `update_color.php`, `get_events.php`, `get_caller_data.php`, `log_phone_view.php`, `get_phones.php`) будут объединены в единую точку входа `api.php`. Этот файл будет анализировать параметр `action` в запросе и вызывать соответствующий метод сервисного класса.
    *   **Сервисная логика:**
        *   Функция `formatPhoneNumber` из `get_phones.php` будет перенесена в класс-хелпер `Utils.php`.
        *   Логика проверки пользователя из `log_phone_view.php` будет инкапсулирована в метод `logPhoneView` класса `CallerService`.
    *   `dashboard.php`: Будет использовать `AuthService` и `CallerService` для первоначальной загрузки данных.

**Пример структуры файлов PHP:**
```
.
├── config.php
├── api.php
├── dashboard.php
├── php/
│   ├── DatabaseManager.php
│   ├── CommentService.php
│   ├── CallerService.php
│   ├── AuthService.php
│   ├── EventLogger.php
│   └── Utils.php
└── ...
```

## 5. Безопасность (уточнение предыдущего плана)

**Цель:** Усиление безопасности приложения, защита от распространенных уязвимостей.

**Шаги:**

*   **Валидация и санитизация:** Все входные данные (из `$_POST`, `$_GET`, `php://input`) должны проходить валидацию и санитизацию на сервере. Использовать `filter_var` с соответствующими фильтрами или регулярные выражения.
*   **Подготовленные выражения:** Уже используются в `add_comment.php` и `update_color.php`, продолжить их использование для всех запросов к БД.
*   **CSRF-токены:** Внедрить CSRF-токены для всех форм и AJAX-запросов, изменяющих состояние на сервере (например, добавление комментария, изменение цвета). Токен генерируется на сервере, передается на клиент и проверяется при каждом запросе.
*   **Аутентификация/Авторизация:** Централизовать логику аутентификации в `AuthService`. Проверять права доступа пользователя перед выполнением любых действий.
*   **Логирование ошибок:** Улучшить логирование ошибок, используя `error_log` с более подробной информацией, чтобы облегчить отслеживание и устранение проблем безопасности.

## 7. Описание реализованных классов и методов

### 7.1. `AjaxHandler` (файл `ajax_handler.php`)

Единый класс для обработки всех AJAX-запросов от клиентской части.

**Свойства:**
*   `private $pdo`: Экземпляр объекта PDO для работы с базой данных.
*   `private $config`: Массив с конфигурацией приложения.

**Методы:**
*   `__construct($pdo, $config)`: Конструктор класса, принимает подключение к БД и конфигурацию.
*   `handleRequest($action)`: Роутер. Принимает строку `action` и вызывает соответствующий метод класса.
*   `sendSuccess($data = [])`: Отправляет успешный JSON-ответ клиенту.
*   `sendError($message, $statusCode = 400)`: Отправляет JSON-ответ с ошибкой.
*   `formatPhoneNumber(?string $phone_string)`: Вспомогательный приватный метод для форматирования телефонных номеров.
*   `get_phones()`: Обработчик для `action=get_phones`. Получает и возвращает отформатированные номера телефонов для указанного `id`.
*   `add_comment()`: Обработчик для `action=add_comment`. Добавляет комментарий и, при необходимости, обновляет цвет строки.
*   `update_color()`: Обработчик для `action=update_color`. Обновляет цвет строки и добавляет соответствующее событие.
*   `log_phone_view()`: Обработчик для `action=log_phone_view`. Логирует событие просмотра телефона, избегая дублирования от одного и того же пользователя подряд.
*   `get_caller_data()`: Обработчик для `action=get_caller_data`. Возвращает основной список записей для таблицы Tabulator.
*   `get_events()`: Обработчик для `action=get_events`. Возвращает историю событий для указанного `id`.
